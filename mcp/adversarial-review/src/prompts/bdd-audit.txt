# Discipline

1. Read the feature files and the step implementations directly. Do not infer alignment from naming.
2. A step that exists is not a step that is wired — trace each Gherkin step to its runtime handler.
3. A step that runs is not a step that asserts — flag steps whose only body is a log/print/no-op.
4. If the user passed phase-skip flags via args, honor them. Otherwise run all three phases.
5. If a step talks to a layer it shouldn't touch (e.g. UI scenarios driving the database directly when the architectural intent has them go through the API layer), flag it.
